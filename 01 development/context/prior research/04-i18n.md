# Chapter 04 — i18n / multi-language per page

R032 ships the **per-page localization layer**: an operator can author a
page in English and add French / Spanish / etc. variants; the storefront
serves the right one based on URL prefix and `Accept-Language`, falling
back to the default with a "this page hasn't been translated" banner.

## Schema shape (host integration)

R032 ships a **pure helper library**. The host page integrates by adding
an optional `locales?: LocalePageMap` field on `EditorPage`:

```ts
interface LocalePageMap {
  defaultLocale: string;                       // e.g. "en"
  locales: Record<string, LocalizedPage>;      // keyed by BCP-47 tag
}

interface LocalizedPage {
  tree: Block[];
  meta?: { title?: string; description?: string; … };
  updatedAt?: number;
  source?: "human" | "auto" | "machine-paste";
}
```

When `locales` is undefined, pages render exactly as before — i18n is
purely additive.

## URL routing

Two locale-routing modes coexist:

- **Prefix routing** — `/fr/about` serves the French variant of `/about`.
  Default locale is *unprefixed* (`/about` not `/en/about`) so existing
  public URLs stay stable for SEO. `parseLocalePrefix(pathname, available)`
  returns `{ locale, restPath }` or null if no available locale matches.
- **Accept-Language fallback** — when no URL prefix is present,
  `parseAcceptLanguage(header)` returns a quality-sorted tag list, and
  `resolveLocale` matches each against available locales (exact match
  first, then language-only — `fr-CA` falls back to `fr`).

`resolveLocale({ pathname, acceptLanguage, override }, page)` collapses
all of the above into a single decision with priority:

1. **Override** (cookie / query / user preference)
2. **URL prefix**
3. **Accept-Language** (quality-sorted)
4. **defaultLocale**

The result includes `wasFallback` so the layout can render a banner when
appropriate, and a `source` tag for cache-key derivation.

## Fallback behaviour

`localizedTree(page, locale)` returns the requested tree if present;
otherwise it returns the default tree with `wasFallback=true`. The host
layout is expected to render an inline banner like:

> 🌐 *This page hasn't been translated to French yet — showing the
> English version. [Help us translate →](mailto:…)*

Banner copy is host-side; the helper provides the boolean.

## hreflang emission

`buildHreflangLinks(pathname, page, origin)` returns `<link
rel="alternate" hreflang="…">` tags for every available locale plus
`hreflang="x-default"`. Storefront stamps these into `<head>`. This is
what unblocks Google's locale-aware indexing — without it, French pages
get treated as duplicates of English.

Default locale is referenced both by its BCP-47 tag (`hreflang="en"`)
*and* `hreflang="x-default"` so Google's algorithm has the canonical
fallback marker.

## Auto-translate stub

R032 intentionally does **not** ship a real machine-translation API call.
The "Auto translate" button in the editor is wired to a host-side
endpoint that decides whether to call a real provider (T6 territory) or
return the source unchanged.

`cloneTreeForLocale(tree, opts)` is the pure helper that walks a source
tree and rewrites every visible string prop (text, label, alt,
ariaLabel, description, title, placeholder, subtitle, heading,
buttonText) by passing each through a supplied async `translate`
function. New ids stamped with `_loc` (or operator-supplied suffix) so
the source and locale trees don't share references.

Default identity translator preserves source text verbatim — that is the
**operator-paste path**: clone the English tree, then the operator pastes
French into each block in the editor.

## Locale-completeness audit

The editor sidebar shows a per-locale status badge ("complete" /
"missing 4 strings" / "untranslated"). `auditLocale(sourceTree,
targetTree)` walks both trees in parallel by index (positional, not
id-matched, since cloned ids are suffix-modified) and counts:

```ts
{
  totalStrings: number;          // translatable props in source
  translated: number;            // target string differs from source
  untranslated: number;          // empty or identical to source
  missingBlocks: number;         // present in source, absent in target
  complete: boolean;             // untranslated === 0 && missingBlocks === 0
}
```

The editor surfaces `untranslated > 0` as a yellow badge, `missingBlocks
> 0` as a red badge.

## Out of scope (per prompt)

- **Real translation API** — T6 territory; R032 ships the stub + the
  operator-paste path.
- **RTL layouts** — `dir="rtl"` flag and per-block flip handling are
  R+1; for now Arabic/Hebrew authors get LTR.
- **Per-block locale override** — operator authors entire pages in one
  locale at a time, not mixed-locale within a single page.

## Touch points

- `src/lib/i18n.ts` — pure helpers (this round).
- `EditorPage.locales?` — host adds the field.
- Storefront SSR layout — calls `resolveLocale` early, then passes the
  resolved locale into the block renderer; calls `buildHreflangLinks`
  to stamp into `<head>`.
- Editor topbar — language picker dropdown (host UI).
- Editor sidebar — status badges per locale (host UI).

## What success looks like

An operator with a French audience can ship `/fr/about` without writing
a single line of code or routing. The English version stays at `/about`
unchanged. Google indexes both correctly. When the operator hasn't
translated `/contact` yet, French visitors see the English page with a
single inline banner pointing them home.
