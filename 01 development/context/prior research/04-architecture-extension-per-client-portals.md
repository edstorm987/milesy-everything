# `04` Architecture Extension — Per-client portals + milesymedia stitch

> Authored 2026-05-05 from Ed's directive. Extends `04-architecture.md`
> (the locked v1 architecture) without contradicting it. Where they
> differ, the changes here supersede.

The original architecture treated the portal as **one Next.js app**
running multi-tenant. Operating reality refines this: the portal is
**one shared app for pre-Live work** AND **one custom portal per Live
client**. The marketing site is the front door for both. Plus the
**aqua portal app itself is the editor + generator** for those custom
portals — same engine.

## The three surfaces (revised)

```
milesymedia.com  (THE front door)
├── Public marketing pages              ← static site we already have
├── /login                              ← auth gateway (single origin)
├── /demo                               ← sandboxed demo
└── /portal/...                         ← Aqua portal app
    │
    ├── For agency team                 ← runs the agency
    ├── For pre-Live clients            ← phase progress + onboarding
    └── For Live clients                ← THE EDITOR for their custom
                                          portal (web editor + plugin
                                          installs + brand kit + content)

clients/<slug>/                         ← THE CUSTOM PORTAL per Live client
                                          (separate folder in repo,
                                           generated by Aqua portal,
                                           edited via Aqua portal)
```

## Lifecycle — when a client's custom portal materializes

```
Discovery → Design → Development → Onboarding → Live
                                          │
                                          └─ Aqua portal generates
                                             clients/<slug>/ from the
                                             client's brand kit +
                                             installed plugins +
                                             content. "Export to repo"
                                             button in the editor.

After Live, edits flow through the Aqua portal editor — same web
editor, same plugin installer, same brand-kit panel. Each save flushes
the changes back to clients/<slug>/ via the same git-based promote
flow T3's website-editor already uses.
```

## Rationale (why per-client folders, not per-tenant in shared app)

- **Aqua portal app stays small.** Hundreds of clients = hundreds of
  custom portals; if all the content lived in the shared app's
  storage, the app bloats and queries get slow. Per-client folders
  push the content into per-deployment static + per-client storage.
- **Production isolation.** A bug in one client's portal doesn't take
  the agency or other clients down.
- **Branded build artefact.** Client owns a deployable thing —
  predictable, auditable, exportable, eject-friendly.
- **Same editor, different target.** The Aqua portal's web editor
  writes to either the shared app's storage OR (for Live clients) the
  per-client repo folder. The plugin model and block library are
  identical at every level — it's the storage target that changes.
- **Vercel monorepo stitching.** Vercel treats `04 the final portal/`
  as one project; rewrites stitch milesymedia (root) +
  Aqua portal (`/portal/*`) + each client portal
  (`<client-domain>` or `clients/<slug>/embed`) at the edge.
- **Same auth origin.** All login still terminates at milesymedia.com.
  Cookies are scoped to milesymedia.com. The custom portals iframe
  for performance + sit on their custom domain — but auth round-trips
  to milesymedia.com.

`clients/<slug>/` is itself a Next.js app (or a subset — a Next.js
deployment unit, even if it shares much of `portal/`'s code via
workspace deps). Each is self-contained enough to deploy
independently. `clients/<slug>/portal-config.json` (or similar)
captures the brand kit + installed plugins + content tree;
`clients/<slug>/src/app/...` is the materialized site.

## "Export to repo" flow

When a Live client's portal is ready to ship as their own deployable:

1. **Agency operator clicks "Export to repo"** in the Aqua portal
   editor.
2. **Aqua portal collects:**
   - Brand kit (logos, colours, fonts, customCSS)
   - Installed plugins (with their per-client config)
   - Active portal variants per `PortalRole` (login, account, etc.)
   - All editor pages + block trees
   - Custom domain config
3. **Materialize:**
   - Create `clients/<slug>/` if not present.
   - Write a Next.js shell that imports plugin packages (same
     workspace deps as the shared portal).
   - Write per-page route files using the editor's published block
     trees.
   - Write a `portal-config.json` mirror.
   - Write a `vercel.json` with the right rewrites + custom domain.
4. **Commit + push** to a branch; open a PR (lifts T3's existing
   GitHub PR promote flow).
5. **On merge** Vercel deploys it. Custom domain points at the new
   deployment.

After export, edits in the Aqua portal editor write to the per-client
repo folder (via the same promote mechanism), so the editor remains
the single control surface — the "git-based web editor" Ed described.

## Iframe-embedded mode (performance)

For end-customer surfaces (Felicia's shoppers logging into her account
view), the per-client portal is iframe-embedded into the client's own
website (luvandker.com hosts an iframe pointing at
`<client-deployment>/embed/login`). Same auth origin
(milesymedia.com cookies) so the session works in both surfaces.

Vercel's edge stitching keeps the network hop low — when a visitor on
luvandker.com loads the iframe, Vercel routes to the per-client
deployment in the same project, no cross-region latency.

## Preset portals

Same shape as the phase-preset plugin bundles, but for the materialized
portal:

- **Service portal preset** — agency-team-driven service brief +
  deliverable feedback loop.
- **Skincare brand preset** — Felicia's pattern (storefront + members +
  affiliates + journal).
- **Membership-only preset** — gated content + recurring billing +
  member directory.
- **Affiliate-only preset** — referral dashboard + payout management.
- ...

Each preset is a JSON manifest:
```ts
{
  id: "skincare-brand",
  label: "Skincare brand",
  description: "...",
  installedPlugins: ["website-editor", "ecommerce", "memberships", "affiliates", "client-crm"],
  portalVariants: { login: "...", account: "...", orders: "...", affiliates: "..." },
  starterContent: { ... },
  defaultBrand: { ... },
}
```

When a Live phase activates and "Export to repo" runs, the operator
picks a preset and the materialization seeds from it.

## Cookie / auth surface (clarified)

Single-cookie origin stays milesymedia.com. The cookie payload now
optionally carries `portalSlug` for Live clients — when set, the
agency-side admin treats this as "you're editing client X's custom
portal" rather than working in shared storage.

```
SessionPayload {
  userId, role, agencyId,
  clientId?,      // for client-* roles
  portalSlug?,    // when admin is operating in per-client portal context
  isDemo?,        // sandbox marker (R4)
}
```

## What does NOT change

- The plugin model (Aqua manifest plugins).
- The role hierarchy (7 roles).
- The phase lifecycle (6 default phases).
- The brand-kit-as-CSS-vars pattern.
- The pool model database scoping (each row carries
  `agencyId` / `clientId`).
- The single-cookie auth.
- The web editor + 58 blocks + variant subsystem.

The architectural extension is **about deployment topology**, not
about replacing any of the v1 mechanics.
